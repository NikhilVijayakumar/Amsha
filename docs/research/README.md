# Amsha Research Package

This directory contains the necessary components, baselines, and experiment scripts to empirically validate the Amsha framework, specifically aligning with the requirements detailed in the `docs/paper/user_inputs/app_execution_results.md` document.

## Overview

The `research` package is intentionally kept parallel to the main `src` directory to preserve structural separation between the core application and research/evaluation scripts, while remaining easily importable within the project space.

### Directory Structure

- **`research/baselines/`**: Implementations of alternative methods or wrapper scripts for existing baselines (e.g., vanilla CrewAI, AutoGen) for comparative performance testing.
- **`research/datasets/`**: Any benchmarking data, custom logs, or dataset downloader scripts required to feed input into the experiments.
- **`research/experiments/`**: Standalone, reproducible Python scripts mapped directly to the sections described in the paper's execution results plan.

## Setup and Installation

The research scripts depend on heavy data science and machine learning packages (e.g., `scikit-learn`, `matplotlib`, `seaborn`, `jupyter`) which are not required for standard Amsha deployment. These are managed via an optional dependency group in the main `pyproject.toml`.

To install Amsha with all necessary research dependencies, run the following command from the root of the project inside your virtual environment:

```bash
pip install -e .[research]
```

## Execution and Configuration

The research framework is driven by a central configuration file and a unified experiment runner, allowing you to seamlessly switch between cloud APIs and locally hosted models.

### Configuration (`research/config.yaml`)

Edit the `research/config.yaml` file to define your active models, endpoints, and environment variables for API keys.
- **Providers Supported**: Google (Gemini), Azure (OpenAI), OpenRouter (Free Tier), and LM Studio (Local).
- API keys should be set in your `.env` file (e.g. `GEMINI_API_KEY`, `AZURE_OPENAI_API_KEY`).

### Using the Unified Runner

All experiments are executed via the `research.runner` module. Because querying large models can be time-consuming or hit rate limits (and local models must be run sequentially), the runner saves incremental JSON outputs into the `research/results/` directory. 

You can aggregate these individual runs at any point.

#### 1. Run All Cloud Models
To run a specific experiment against all models marked as `tier: cloud` in the config:
```bash
python -m research.runner --experiment 2 --all-remote
```

#### 2. Run a Specific Local Model
If you have LM Studio running a specific model (e.g., Qwen), you can target it directly by the name defined in your config:
```bash
python -m research.runner --experiment 2 --model "qwen3-14b"
```

#### 3. Run All Experiments for a Model
```bash
python -m research.runner --experiment all --model "qwen3-14b"
```

#### 4. Aggregate Results
Once you have run the experiments across the disparate models/tiers you need, aggregate the saved JSON chunks into a master dataset:
```bash
python -m research.runner --aggregate
```
This will generate `research/results/MASTER_AGGREGATION.json`, which contains the compiled empirical data needed to fill in the `📝 YOUR DATA` sections of your paper.

## Sub-Experiments Mapping

1. `1_crew_construction` - CrewBuilder assembly overhead & equivalence.
2. `2_llm_performance` - Latency, parameter sensitivity, and cost tradeoffs.
3. `3_monitoring_observability` - Crew Monitor overhead bounds & feature consensus.
4. `4_evaluation_pipeline` - JSON sanitization & cross-evaluator agreement.
5. `5_end_to_end` - Full pipeline timing and comparative analysis.
6. `6_privacy_verification` - Telemetry neutralization and air-gapped guarantees.
